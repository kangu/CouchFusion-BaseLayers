# Initial Prompt
Revent the last change to add the prefixes to the public runtime, I don't want those exposed. Instead, keep SSR code as it is, but for client side navigation, when the content call returns 404, just let the normal flow of page rendering continue.

# Implementation Summary
Removed the public runtime ignore mirroring and adjusted the content middleware to treat client-side 404s as non-blocking: if a content fetch returns 404 on the client, the middleware now exits without aborting navigation, while SSR keeps the existing 404 behavior.

# Documentation Overview
- `layers/content/app/middleware/content.global.ts` no longer relies on public runtime ignores; on client navigation, a 404 from content fetch simply returns, allowing the normal route to render.
- Public runtime exposure of ignore lists was reverted to keep prefixes private; server/runtime ignores remain in `runtimeConfig.content.ignore`.

# Implementation Examples
- Visiting `/members/jobs` on client navigation: content fetch 404 is swallowed, so the route renders normally; SSR still aborts with 404 for genuine missing content pages.
