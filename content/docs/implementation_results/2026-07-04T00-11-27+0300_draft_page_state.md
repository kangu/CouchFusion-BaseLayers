# Draft Page State

## Initial Prompt

Add draft state support to content pages so work-in-progress pages are available only to admins/editors, return 404 publicly, are hidden from sitemap output, and can be changed inline from the admin page list. The state applies across all locales for a page.

## Plan

1. Store `publicationState` on the master content page document.
2. Normalize missing/legacy state to `published` and reject invalid mutation payloads.
3. Gate draft reads in the server page API by admin/editor roles.
4. Add a dedicated publication-state update endpoint that updates only the master document.
5. Reflect state in the page store and admin pages UI.
6. Exclude draft page groups from sitemap generation.

## Implementation Summary

- Introduced `ContentPagePublicationState` and normalized `publicationState` through content document conversion, page summaries, and API responses.
- Added content-editor session resolution for read access to drafts, allowing `admin`, `_admin`, and `editor`.
- Added `PATCH /api/content/pages/publication-state` with strict state validation and master-document updates.
- Updated the Pinia pages store with `updatePublicationState()` and draft defaults for new admin-created pages.
- Added status filtering and inline status updates to the admin pages list.
- Updated sitemap generation to remove draft content routes and localized members of draft page groups.
- Added focused coverage in `content/tests/api-content-pages.spec.ts` and `sitemap-xml/tests/sitemap-draft-pages.spec.ts`.
