# Nostr Wallet Connect Provider

## What changed

- Added the `nwc` Lightning provider alongside Strike, Alby PAT, and Blink.
- Added CouchDB `_config` resolution for a raw `nwc_connection_uri` and optional reconciliation interval.
- Added an adapter around `@getalby/sdk`, keeping SDK types and secret-bearing values out of the public Lightning API.
- Added provider-status indexing for normalized invoice documents, an NWC-safe generic-expiry exclusion, verified refresh processing, and a single-instance subscription/reconciliation Nitro plugin.

## Setup

1. Create a receive-only NWC app connection in Alby Hub or another NIP-47-compatible wallet. Grant only invoice creation, invoice lookup, and received-payment-notification capabilities. Do not grant `pay_invoice`, keysend, or administrative permissions.
2. Put the complete NWC URI in the consuming app's CouchDB `_config` section. For an app whose slug is `bitvocation`:

```ini
[cf_env_bitvocation]
lightning_default_provider=nwc
nwc_connection_uri=nostr+walletconnect://<wallet-pubkey>?relay=<encoded-relay-url>&secret=<connection-secret>
nwc_reconcile_interval_ms=60000
```

3. Restrict CouchDB `_config` access. The raw URI includes the Nostr connection secret. The layer reads it only on the server; it must never be copied into Nuxt public runtime config, page data, logs, or client code.
4. Run one Nitro instance in v1. That process owns the NWC notification subscription and reconciliation timer.
5. Use the existing payment routes; no client API changes are required:

```http
POST /api/lightning/purchase
Content-Type: application/json

{ "product": "your_product" }
```

or, for a donation:

```http
POST /api/lightning/donation
Content-Type: application/json

{ "amount": 1000, "memo": "Support" }
```

Both return the normal BOLT11 invoice response. Existing status lookup and authenticated SSE routes report the verified payment lifecycle.

## Operational notes

- NWC invoices use existing `lightning_invoice` documents and fulfilment paths.
- A relay notification never settles an invoice by itself; it triggers an authoritative invoice lookup and amount check.
- Reconciliation checks pending NWC invoices at startup and on the configured interval, recovering payments missed while the app or relay was offline.
- The current test suite uses mocks. Before production enablement, perform a staging checkout with an NWC wallet connection that can create and settle a small invoice.
